# Cuckoo 自动化审计工作流 (Audit Workflow)

> **更新日期**: 2026-05-05  
> 每次开发迭代结束后复用本流程，确保代码稳定性与需求对齐。

---

## 🚀 推荐执行顺序 (Pipeline)

```
1. 语义与需求对齐  →  确认"做对了"
2. 数据流审计      →  找逻辑漏洞
3. 状态机审计      →  找状态死路
4. 边界条件审计    →  找崩溃点
5. 错误处理审计    →  找吞错点
6. 效能路径审计    →  找性能瓶颈
7. 最小修复生成    →  输出 Patch
```

> **核心原则**：每次只跑一个维度，先语义后优化，输出结构化 Markdown。

---

## 📋 审计指令模板

### 一、语义与需求对齐 (Spec Alignment)

```
任务：检查程式码是否符合需求/设计
输入：PRD/设计说明 + 变更的代码（diff）
输出：
1. 与需求不一致的地方（逐条列出）
2. 未覆盖的使用场景
3. 隐含假设（未在设计中说明但代码依赖）
4. 建议修正（含最小修改方案）
```

**用途**：避免"做对的事但做错方式"或"做了不该做的事"

---

### 二、数据流审计 (Data Flow)

```
任务：建立数据流图并找风险
输入：代码
输出：
1. 数据来源（input）
2. 中间处理（transform）
3. 输出（output）
4. 信任边界（external/internal）
5. 潜在问题（未验证输入、过度信任、数据泄漏）
```

**用途**：比单纯安全扫描更容易抓"逻辑层漏洞"

---

### 三、状态机审计 (State Machine)

```
任务：分析状态转移是否完整
输入：代码（含状态字段）
输出：
1. 所有状态与转移图
2. 无法到达的状态
3. 缺失的转移（dead end / 未处理情况）
4. 可能 race condition 的地方
```

**用途**：订单、支付、workflow 系统必用

---

### 四、边界条件审计 (Edge Cases)

```
任务：找出边界与极端情况
输入：代码
输出：
1. 所有边界条件（null, empty, 0, max, NaN）
2. 未处理的情况
3. 可能崩溃或错误输出的点
4. 建议测试案例（含输入与预期输出）
```

**用途**：快速补齐测试覆盖

---

### 五、复杂度与可维护性 (Complexity)

```
任务：评估可读性与维护成本
输入：代码
输出：
1. 高复杂函数（圈复杂度）
2. 重复逻辑
3. 过度耦合
4. 可抽象/拆分的地方（附重构建议）
```

**用途**：避免技术债累积

---

### 六、依赖与副作用审计 (Side Effects)

```
任务：找出隐性副作用
输入：代码
输出：
1. 所有副作用（IO、DB、global state）
2. 隐式依赖（未明确传入的变量/服务）
3. 测试困难点
4. 建议改为纯函数的部分
```

**用途**：让系统更可测、更稳定

---

### 七、错误处理审计 (Error Handling)

```
任务：检查错误处理完整性
输入：代码
输出：
1. 所有可能抛错的点
2. 未捕捉或吞错的地方
3. 不一致的错误处理模式
4. 建议统一策略（retry / fallback / log）
```

**用途**：提升可靠性

---

### 八、效能路径审计 (Hot Path)

```
任务：找出性能瓶颈
输入：代码
输出：
1. 热路径（高频执行）
2. 高成本操作（IO / 计算）
3. 可优化点（cache / batching / lazy）
4. 预估影响（大致量级）
```

**用途**：不需要 profiler 也能先抓大头

---

### 九、测试覆盖审计 (Test Coverage)

```
任务：评估测试是否充分
输入：代码 + 测试
输出：
1. 未覆盖的功能
2. 缺失的 edge cases
3. 不稳定测试（flaky）
4. 建议新增测试（具体案例）
```

---

### 十、API 合约审计 (Contract)

```
任务：检查 API 一致性
输入：代码（API 层）
输出：
1. 请求/响应 schema
2. 不一致命名
3. 破坏向后兼容的变更
4. 建议标准化方案
```

---

### 十一、资源生命周期审计 (Resource Lifecycle)

```
任务：检查资源是否正确释放
输入：代码
输出：
1. 建立/使用/释放流程
2. 潜在 memory leak / connection leak
3. 建议改进（pooling / context）
```

---

### 十二、最小修复指令 (Patch Generator)

```
任务：只输出最小修复
输入：代码 + 问题描述
输出：
- 仅包含必要修改的 diff
- 不改动无关逻辑
```

**用途**：直接接 CI 自动修

---

## 🛠 Cuckoo 系统审计检查点

| 维度 | 在 Cuckoo 中的重点检查区域 |
|------|---------------------------|
| 语义对齐 | `recipes-page.tsx` 删除逻辑 vs `database.rs` 软删除 |
| 数据流 | `report_telemetry` URL 注入、`TelemetryPayload` 数据泄漏 |
| 状态机 | 订单状态流转（draft→submitted→in_progress→ready→completed）|
| 边界条件 | 库存 qty=0、配方递归嵌套深度、`parseSafeFloat` NaN |
| 错误处理 | 140 个 `lock().unwrap()`、前端 `invoke` 无 try-catch |
| 效能路径 | `calculate_recipe_cost` 递归、`get_inventory_summary` 子查询 |
| 副作用 | `finish_ticket` 同时扣库存 + 改订单状态（多步非事务）|
| 资源 | `Mutex<Connection>` 锁竞争、打印串口连接释放 |
| API 合约 | `confirm()` vs `AlertDialog`、命名不一致（camelCase vs snake_case）|

---

## 📝 审计输出规范

每次审计结束后：

1. **审计报告**存入 `docs/` 目录，命名格式：`{type}-audit-{date}.md`
2. **发现的问题**分级后追加到 `docs/backlog-and-fix-list.md`
3. **已修复项**在 `TODOs.md` 中标记 `[x]`

---

*本工作流在每次版本开发结束后复用，确保审计驱动开发的一致性。*
