# Cuckoo 中小型餐廳業務流程與系統適配審計報告

> **審計日期**: 2026-04-28  
> **目標客群**: 中國本土小型餐廳（未來有向多店鋪/中央廚房擴張的需求）  
> **重點範圍**: 端到端業務流程適配度、未來多店擴張可行性、防呆與易用性  

---

## 一、 用戶核心業務流程梳理 (User Workflow)

根據中國小型餐廳的實際運營場景，系統必須滿足以下全業務生命週期的閉環：

### 1. 門店營銷與履約流程 (Order & Fulfillment)
*   **前台接單**: 客戶到店或線上點餐 → 收銀員確認需求 → 手動在 POS 端輸入並提交。
*   **出單分配**:
    *   **廚房餐單**: KDS 看板同步顯示，或自動打印機出「廚房單」（需有加料、去料、特殊備註），指導後廚生產。
    *   **前台訂單**: 前台打印「點餐小票/杯貼」，包含取餐號。前台依據訂單狀態進行**叫號取餐**或整理打包交給**聚合配送平台**（美團/餓了麼/達達）騎手。

### 2. 供應鏈與進銷存流程 (Supply Chain & Inventory)
*   **自動扣料**: 訂單完成後，系統根據商品關聯的「配方（Recipe）」與「子配方（半成品）」，自動精準扣減對應材料的庫存。
*   **成本與耗損可視化**:
    *   每天日結時，店長能透過報表查看當天消耗了多少斤肉、多少克菜。
    *   根據進貨的「批次價格」，動態估算當天營業額對應的「真實食材成本」與「毛利」。
    *   支援手動錄入廚房意外損耗（如掉落、變質），確保帳實相符。
*   **預警與採購**: 庫管設定安全庫存（警戒線）。當材料庫存低於警戒線時，系統紅燈預警，庫管一鍵生成「採購單」發給供應商。

### 3. 未來擴展流程：多店與中央廚房 (Multi-Store & Central Kitchen)
*   **中央生產**: 單一廚房（中央廚房）大量生產半成品（如：預製滷肉、切配好淨菜）。
*   **配送與二次加工**: 半成品配送至各個前置門店，門店接單後只做簡單的加熱或二次組裝（組裝也依賴配方扣料）即出餐。

## 二、 目前系統適配度審計 (Adaptability Audit)

基於目前 Cuckoo v1.2.0 (Tauri + SQLite 本地優先架構) 的現狀，適配度評估如下：

### ✅ 完美適配的模塊 (Ready)
1.  **自動扣料與配方樹**: 目前已支持 `原材料 -> 半成品(子配方) -> 最終商品` 的多級配方樹，且訂單提交時會自動深度遍歷扣料，完美契合二次加工需求。
2.  **批次成本計算**: 採購單入庫自動生成批次，庫存採用批次管理，成本核算基於批次價格，可精確計算毛利。
3.  **預警機制**: 已實現低庫存閾值設定與 Dashboard 紅框預警。
4.  **半成品生產**: 已具備「生產單 (Production Orders)」模塊，支持通過半成品配方投料，轉化為半成品庫存。

### ⚠️ 部分缺失或需優化的模塊 (Needs Optimization)
1.  **前台聚合配送與叫號 (履約端)**:
    *   **現狀**: 目前有訂單列表，但缺乏專門面向「聚合配送（外賣/打包）」的狀態流轉（如：待接單、待騎手取貨）。
    *   **防呆問題**: 點餐時缺乏取餐號自動生成邏輯，前台僅靠訂單號（一長串數字）難以叫號。
2.  **損耗與日報表 (數據端)**:
    *   **現狀**: 有「原料消耗報表」，也有「庫存盤點」模塊。
    *   **防呆問題**: 報表缺乏「理論消耗」與「實際消耗（含報損）」的對比視圖。非工程人員很難一眼看出「今天浪費了多少錢」。

### ❌ 架構級不匹配：多店與雲端同步 (Major Blocker)
*   **現狀**: Cuckoo 是基於本地 SQLite 運行的桌面客戶端（Local-first）。
*   **衝突點**: 多店鋪和中央廚房配送**必須**依賴雲端數據庫（Cloud-sync 或 SaaS 架構）來實現跨店庫存調撥與全局報表。
*   **解決方案評估**: 
    1. 若維持本地架構：無法實現實時多店。
    2. 若要滿足未來需求：必須在 v2.0 引入遠端數據庫（如 Supabase / PostgreSQL），或者實現基於雲的數據同步機制（Local-first with Cloud Sync），並增加 `branch_id`（門店 ID）字段。

## 三、 防呆與易用性專項審計 (UX/Fool-proofing)

針對中國小型餐飲從業人員（可能年紀較大、非 IT 專業），目前界面存在以下「不夠防呆」的痛點：

1.  **「黑洞輸入」危險**: 採購單、盤點單的數量輸入框如果填錯（如輸入文字），系統會靜默變為 0 寫入數據庫（已被識別，待修復）。
2.  **繁簡體混亂**: 系統介面混雜繁體中文（如 `儲存`, `載入訂單失敗`），增加理解成本。
3.  **缺乏快捷鍵**: POS 收銀在高峰期極度依賴鼠標，沒有配置如 `Space` 結帳、`Esc` 清空購物車等鍵盤防呆快捷鍵。
4.  **警告不夠「視覺化」**: 庫存低於警戒線時，雖然 Dashboard 有提示，但在 POS 點單時沒有「庫存不足」的直觀阻斷，容易導致超賣（雖然底層有負庫存防護，但前台感知弱）。

## 四、 端到端測試流程設計 (E2E Testing Flow)

為了驗證系統能跑通上述核心流程，設計以下標準測試 SOP：

### 🧪 測試場景：從採購到銷售的完整生命週期

**前置準備**
1. 建立材料：`豬肉` (kg，安全庫存 5kg), `滷包` (包), `滷肉汁` (升 - 半成品)。
2. 建立配方：
   * 半成品配方：`一鍋滷肉汁` = 10kg 豬肉 + 2個滷包。
   * 成品配方：`滷肉飯` = 0.2升 滷肉汁。
3. 建立商品：`招牌滷肉飯`，綁定上述成品配方。

**Step 1: 供應鏈觸發 (庫管)**
*   **操作**: 查看 Dashboard 發現豬肉庫存為 0（低於 5kg 警戒）。新建採購單，買入 20kg 豬肉（100元）和 10個滷包（10元）。
*   **預期**: 採購單確認並「入庫」後，庫存增加，批次成本自動生成。

**Step 2: 中央生產 (廚房)**
*   **操作**: 建立生產單，生產 1 鍋 `滷肉汁`（產出 10升）。
*   **預期**: 扣除 10kg 豬肉和 2個滷包的庫存，增加 10升 `滷肉汁` 的庫存。成本自動累加到滷肉汁上。

**Step 3: 門店營銷 (收銀)**
*   **操作**: 進入 POS，點選 2 份 `招牌滷肉飯`，備註「多汁」，提交訂單。
*   **預期**: 訂單狀態為「待處理」。

**Step 4: 廚房履約 (廚師)**
*   **操作**: 進入 KDS 頁面，看到 2 份滷肉飯的工單。點擊「完成」。
*   **預期**: 訂單狀態變為「已完成」。

**Step 5: 成本與扣料驗證 (店長)**
*   **操作**: 檢查庫存，確認 `滷肉汁` 減少了 0.4升。
*   **操作**: 進入報表頁面，查看當天毛利與消耗。
*   **預期**: 毛利計算正確（營收 - 0.4升滷肉汁的成本），報表無報錯。
