# 配方管理审计与 TODO

最后更新：2026-05-05

## 范围

本轮审计覆盖以下完整链路：

1. 注册原材料
2. 建立成本基线
3. 组装半成品
4. 追加其他原材料或子配方
5. 组装成品
6. 绑定菜单并观察成本与毛利

同时覆盖以下横向能力：

- 配方类型管理
- 删除安全
- 批量编辑
- 配方、库存、菜单之间的 UI/UX 连续性

## 当前结论

### P0 结构问题

- 当前配方页仍然是“表格管理页”，不是“业务路径页”。
- 用户看不到“材料 -> 半成品 -> 成品 -> 菜单”的明确引导。
- 成本计算能算出来，但成本来源没有被清晰讲明。
- 配方类型过去是硬编码字符串，已经改成主数据，但后续工作流还没完全接上。

### P1 体验问题

- 原材料、批次、配方、菜单分散在多个页面，缺少顺滑交接。
- 示例数据入口之前依赖整页刷新，且后端种子逻辑存在硬编码 ID 风险。
- 删除、不可删、为什么删不了、删了影响谁，系统提示并不一致。
- 当前列表可以单条编辑，但仍然缺少批量编辑和批量状态切换。

### P1 业务问题

- 半成品缺少完整的“产出物 / 产出单位 / 产出状态”建模闭环。
- 菜单虽然能绑定配方，但页面没有清楚展示“菜单只绑定配方，不直接绑定原材料”的业务关系。
- 配方项目前主要是 `material` 和 `sub_recipe` 两种 UI 表达，扩展位还没有真正产品化。

### P2 管理问题

- 缺少统一依赖视图，无法一眼看出“某个类型 / 配方 / 材料被谁使用”。
- 示例数据、类型维护、配方维护目前都是散点能力，缺少统一治理视角。

## 设计方向

### 目标心智模型

系统应该明确教会用户：

1. 先定义材料
2. 再补库存批次与成本
3. 然后做半成品
4. 再把半成品与其他材料组装成成品
5. 最后把成品绑定菜单，并持续观察成本与毛利

### 配方类型目标

配方类型必须支持：

- 可选
- 可改
- 可加
- 可删

删除时必须有依赖保护：

- 如果仍有配方在用，应阻止删除并提示依赖来源
- 如果是系统类型，应禁止直接删除

### 删除与批量治理目标

统一采用以下原则：

- 先告诉用户谁依赖它
- 再允许删除或软删除
- 无法删除时要给出明确原因
- 列表页逐步补齐批量编辑能力

## UX 优化重点

### 1. 从“资料页”改成“任务流”

当前很多页面还是典型后台资料页心智：

- 先看到一张表
- 再自己推测下一步做什么

更合理的方向应该是任务流：

- 菜单产品未绑定配方时，直接显示“加入配方清单”
- 配方已创建但没明细时，直接显示“开始完善”
- 配方已有明细但未绑定菜单时，直接显示“去绑定菜单”

这类上下文动作比单纯新增按钮更有效。

### 2. 给用户持续的“完成度反馈”

配方工作流不该只有“有/没有”。
建议后续统一成可见状态：

- 待完善：没有任何明细项
- 已建结构：已有明细，但未完成成本核对
- 已可出品：已绑定菜单且成本可计算

这样店长、研发、实施人员看到的是同一套进度语言。

### 3. 强化跨页面交接

当前系统最大 UX 损耗不在单页内部，而在页面切换之间：

- 材料页建完材料，不知道下一步去批次页
- 批次录完成本，不知道下一步去配方页
- 菜单产品缺配方，要靠人自己发现

后续应逐步补：

- 从材料页跳“查看被哪些配方使用”
- 从菜单页跳“补配方”
- 从配方页跳“去绑定菜单”

### 4. 优先做“下一步动作”而不是“更多字段”

业务系统常见误区是补字段比补动作快，但用户更需要的是下一步提示。
所以后续优先级建议是：

1. 引导动作
2. 状态可见
3. 依赖透明
4. 批量治理
5. 复杂字段扩展

### 5. 把风险提示前置

对这类系统，真正的好 UX 不是动画和卡片，而是：

- 修改前告诉用户会影响谁
- 删除前告诉用户为什么删不了
- 批量前告诉用户会改多少条

这会直接减少脏数据和培训成本。

## 五大方向执行规划

### 方向一：把“下一步”做成显式动作

目标：

- 用户在每个节点都知道下一步做什么
- 减少“我接下来该去哪一页”的犹豫

已完成：

- 菜单产品无配方：加入配方清单
- 配方无明细：开始完善
- 配方可出品但未绑定菜单：去绑定菜单

下一步 TODO：

- [x] 配方有明细但未绑定菜单：增加“去绑定菜单”
- [x] 配方成本缺少来源：增加“去补批次成本”
- [ ] 材料刚创建完成：增加“去录入批次”

验收标准：

- 用户从任一入口进入后，最多 1 次点击就能进入下一步

### 方向二：统一流程状态

目标：

- 让店长、研发、实施看到同一套状态语言
- 让筛选、批量和提醒有统一依据

建议状态：

- 待完善：没有任何明细项
- 待核对：已有明细，但成本来源不完整或未核对
- 可出品：结构完整且成本可计算
- 已绑定菜单：已被菜单商品引用

下一步 TODO：

- [x] 设计前端状态计算规则，不新增数据库字段
- [x] 在配方列表加状态列
- [x] 在配方详情页显示当前状态说明
- [ ] 补状态筛选与跨页面统一状态展示

验收标准：

- 任一配方都能在列表中被归入一个明确状态

### 方向三：跨页面接力

目标：

- 把材料、批次、配方、菜单串成连续路径
- 降低跨页后的迷失感

下一步 TODO：

- [x] 菜单页增加“补配方”跳转
- [x] 配方页增加“去绑定菜单”跳转
- [x] 材料页增加“查看被哪些配方使用”
- [x] 批次页增加“影响哪些配方成本”的入口

验收标准：

- 四个核心页面之间至少形成 3 条可见的双向业务跳转

### 方向四：依赖关系透明

目标：

- 修改前知道影响谁
- 删除前知道为什么删不了

下一步 TODO：

- [ ] 配方详情显示：被几个菜单使用、被几个配方引用
- [ ] 配方类型显示：有多少活跃配方在用
- [ ] 材料页显示：被多少配方使用
- [ ] 删除失败统一展示依赖说明

验收标准：

- 高风险对象在删除或修改前都能看到影响范围

### 方向五：批量治理能力

目标：

- 解决高频重复劳动
- 降低人工批量修正的出错率

下一步 TODO：

- [ ] 配方列表支持批量切换类型
- [ ] 配方列表支持批量切换状态或审核标记
- [ ] 材料列表支持批量改分类、最小库存、启停状态
- [ ] 批量操作前显示将影响多少条记录

验收标准：

- 高频管理动作不再要求逐条编辑
