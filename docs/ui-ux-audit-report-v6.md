# Cuckoo 對抗性全局審計報告 v6.0 (多角色業務流程與本地化專項)

> **審計日期**: 2026-04-28  
> **重點範圍**: 跨角色業務流程準確性、防呆設計(Fool-proofing)、繁簡體一致性  
> **審計方法**: 多視角路徑追蹤 + 靜態文本模式掃描

---

## ⚠️ 執行摘要

本次審計切換至真實店鋪運營的 4 個核心角色（店長、收銀、廚師、庫管）視角，發現系統在**「防呆設計」與「本地化（簡體中文強制要求）」上存在嚴重失控**。整個系統充斥著繁簡體混用的情況，且部分核心流程對非工程人員極不友好，存在數據被意外破壞的風險。

---

## 一、繁簡體混用與本地化崩壞 (嚴重)

系統明確要求為「簡體中文程序」，但在代碼掃描中發現大量繁體中文殘留，分佈在側邊欄、Toast 提示、表格狀態與 EmptyState 中，嚴重影響產品的專業度與一致性。

### 🔴 1.1 核心 UI 組件的繁體殘留
- **側邊欄 (Sidebar)**: `AppSidebar` 中存在 `屬性模板`（應為 `属性模板`）。
- **空狀態 (EmptyState)**: KDS 頁面顯示 `暫無待處理工單`（應為 `暂无待处理工单`），而其他頁面顯示 `暂无订单`，組件間不統一。
- **狀態徽章 (Badge)**: `purchase-orders-page.tsx` 中採購單狀態渲染為 `已確認`（應為 `已确认`）。
- **下拉佔位符 (Select)**: 大量表單中存在 `選擇材料`、`選擇供應商`（應為 `选择...`）。

### 🔴 1.2 Toast 報錯提示的繁體殘留
`inventory-page.tsx`、`print-tickets-page.tsx` 及 `useAppActions.ts` 中的報錯提示幾乎全為繁體：
- `載入訂單失敗` -> `加载订单失败`
- `數量格式錯誤，請輸入有效數字` -> `数量格式错误，请输入有效数字`
- `請填寫必填欄位` -> `请填写必填字段`

**建議**: 建立統一的 `i18n` 字典文件或進行全局的正則替換，徹底清洗繁體字元。

## 二、防呆設計與非工程人員易用性 (高風險)

非工程人員（如餐廳收銀員、倉庫大叔）在操作時容易發生誤觸和非標準輸入，系統目前缺乏足夠的防禦機制。

### 🟠 2.1 靜默的破壞性輸入 (Silent Data Corruption)
在採購單、生產單與盤點單中，數量與成本輸入廣泛使用了 `parseFloat(val) || 0` 的寫法。
- **場景**: 庫管大叔在採購成本輸入 `12元`（帶了單位文字）。
- **結果**: `parseFloat` 報錯返回 `NaN`，系統靜默將其轉為 `0` 並存入數據庫。
- **危害**: 沒有報錯彈窗，財務報表會憑空多出一筆 0 元進貨的記錄，這在業務上是致命的。

### 🟠 2.2 毀滅性操作缺乏二次確認 (Missing Confirmations)
- **問題**: `useAppActions.ts` 中的 `delete_recipe_item`（刪除配方明細）等部分刪除操作，點擊後會**瞬間執行**並同步數據庫，沒有 `ConfirmDialog` 攔截。
- **危害**: 廚師在編輯複雜配方時，一旦手滑點錯，配方數據瞬間丟失且無法撤銷。

### 🟡 2.3 購物車缺乏「一鍵清空」防呆
- **場景**: 顧客點了 20 杯飲料，突然說不要了。
- **問題**: `pos-page.tsx` 沒有「清空購物車」的按鈕，收銀員必須點擊 20 次垃圾桶圖標才能清空，這在高並發的點餐高峰期是極糟的 UX。

## 三、跨角色業務流程斷層 (Business Logic Gaps)

### 🟠 3.1 收銀員視角 (Cashier)
- **問題**: `useAppActions.ts` 導出的 `create_order` 方法（用於後台手動建單）中，**硬編碼了 `dine_type: "dine_in"`** 和 `table_no: null`。
- **影響**: 如果系統允許在 POS 以外的頁面（如訂單管理頁）快速建單，這些訂單永遠只能是「堂食」，收銀員無法為外送或外帶手動建單，業務流程被鎖死。

### 🟠 3.2 店長視角 (Manager)
- **問題**: 在 `reports-page.tsx` 下載 CSV 報表時，表頭被硬編碼為繁體字：`["商品", "銷售額", "數量", "均價"]`。
- **影響**: 店長在 Excel 裡打開報表時，會發現系統 UI 是簡體，導出的報表卻是繁體，且與財務系統對接時可能因字段名不匹配而失敗。

### 🟡 3.3 庫管視角 (Warehouse)
- **問題**: 在 `inventory-page.tsx` 中進行庫存「調整」或「報損」時，如果輸入負數（如 `-5`），系統並未嚴格校驗其絕對值邏輯，可能會與後端 `adjust_inventory` 的加減邏輯發生衝突（例如變成雙重負數扣減）。

## 四、修復行動計畫 (Action Items)

### 📌 階段 1：本地化清洗 (預估 1h)
1. 執行全局的繁體轉簡體正則掃描（針對 `src/pages` 和 `src/components` 中的中文字串）。
2. 將 `AppSidebar`、`EmptyState`、`Toast` 和 CSV 導出表頭統一修改為規範的簡體中文。

### 📌 階段 2：防呆防線構建 (預估 2h)
1. 移除所有的 `parseFloat(val) || 0`，改為嚴格校驗：如果輸入不是有效數字，直接 `return` 並 `toast.error("格式不正确")`。
2. 為所有 `delete_` 前綴的操作（特別是配方明細）補齊 `ConfirmDialog` 組件。
3. 在 `pos-page.tsx` 購物車底部加入「清空购物车」按鈕（帶紅色警示與二次確認）。

### 📌 階段 3：業務邏輯補漏 (預估 1h)
1. 解除 `create_order` 中 `dine_type` 的硬編碼，允許傳參。
2. 增加庫存數量輸入的 `<input min="0">` 屬性限制，從 UI 層攔截負數輸入。
