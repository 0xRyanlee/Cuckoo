# Cuckoo 系統架構演進與配方 UX 設計方案

## 🚀 架構方向評估 (Architecture Assessment)

您提出的架構規劃非常務實且具備前瞻性，完美契合了中國餐飲 SaaS 的演進路徑：

1. **單店/早期階段 (Local-First + 遠端監控)**
   - **優勢**: 實體店鋪最怕斷網。本地 PC 運行 Tauri + 後台服務（本地 SQLite）確保了**斷網依然能順暢點單出單**。
   - **遠程運維**: 透過局域網暴露 API，或配合內網穿透（如 FRP/Ngrok），管理員能遠端監控報錯與日誌。
   - **本地擴展**: 局域網內掃描 9100 端口，直連出單機；未來加幾台收銀副機或廚房 KDS 平板，只需連上同一 WIFI 訪問該 PC 的局域網 IP 即可。

2. **多店/擴張階段 (Cloud SaaS + 小程序)**
   - **平滑過渡**: 當擴張為中央廚房+多門店時，將本地 SQLite 遷移至阿里雲 PostgreSQL，前端代碼幾乎不用大改，只需將 API Base URL 指向雲端。
   - **微信生態**: 透過雲端 API，輕易接出微信小程序給店長看「實時營業額」，甚至後續可以做「顧客掃碼點餐」。

## 👨‍🍳 配方與子配方 UX 優化方案 (Recipe UX Options)

目前的配方介面是「扁平化」的（只能看一層），這對於中央廚房製作半成品（如：買香料 + 豬肉 -> 滷肉汁 -> 滷肉飯）來說，極不直觀，且反覆彈窗修改很容易出錯。

為了達到**防呆、直觀、一目了然**的要求，以下提供三種不同維度的 UX 方案：

### 🌟 方案一：可視化節點工作流 (Node-based Visual Flow - 類似 n8n)

完全借鑒 n8n 或 Unreal Engine 藍圖的節點連線設計。
使用開源庫：`React Flow`。

- **運作邏輯**: 
  - 畫布中央是最終產品（如：招牌滷肉飯）。
  - 左側有一個側邊欄，可以將「原材料」或「半成品」**拖拽**到畫布上。
  - 用滑鼠將材料節點與配方節點**連線**，連線上直接顯示「用量 100g」與「損耗率 5%」。
  - 如果拖入的是「半成品節點」，它自己也可以展開，顯示它是怎麼被其他材料合成的。
- **防呆設計**: 
  - 不能將成品的循環依賴連線（如 A 依賴 B，B 依賴 A，連線時會亮紅燈拒絕）。
  - 直觀的成本熱力圖：最貴的材料節點標紅，一眼看出成本大頭。
- **優缺點**:
  - ✅ **絕對的直觀**：對於非工程人員，看到「線連起來變成一道菜」是最符合直覺的。
  - ✅ **震撼的視覺(Wow Factor)**：在對外展示 SaaS 系統時，這個介面能極大提升產品的高級感。
  - ❌ **開發成本較高**：需要處理畫布的縮放、節點佈局、拖拽邏輯。

### 📊 方案二：可展開的樹狀表格 (Nested Tree-Table / Notion Style)

類似 Notion 或財務軟件的無限層級展開列表。

- **運作邏輯**:
  - 表格中列出配方明細。如果某個明細是「半成品」，它的左側會有一個 `>` 箭頭。
  - 點擊箭頭，該行下方會無縫滑出一個子表格，顯示這個半成品的配方。
  - **Inline-editing (行內編輯)**：拋棄彈窗。點擊數值直接變成輸入框，按 `Enter` 保存。
- **防呆設計**:
  - 當修改了底層半成品的材料，頂層成品的「成本估算」數值會用動畫（綠色/紅色閃爍）實時更新。
- **優缺點**:
  - ✅ **操作效率極高**：不需要滑鼠拖拽，全是鍵盤輸入，對熟練的店長或庫管非常友好。
  - ✅ **工程複雜度適中**：基於現有的 Shadcn Table 稍作封裝即可。
  - ❌ **層級過深時視覺易疲勞**：如果配方嵌套超過 3 層，縮進會讓畫面變得擁擠。

### 📱 方案三：米勒列導航 (Miller Columns / macOS Finder Style)

類似蘋果電腦的 Finder 資料夾多列視圖。

- **運作邏輯**:
  - 畫面分為 3 到 4 欄並排。
  - 第一欄：所有最終菜品。點選「滷肉飯」。
  - 第二欄：顯示「滷肉飯」的配方（米飯、滷肉汁、青菜）。點選「滷肉汁」。
  - 第三欄：顯示「滷肉汁」的配方（豬肉、醬油、香料）。
- **防呆設計**:
  - 用戶永遠知道自己「在什麼位置」，不會在編輯半成品時忘記自己最初是為了改哪道菜。
  - 每一欄的最下方有一個固定的快速添加框：`[選擇材料] [數量] [單位] [↵ 添加]`。
- **優缺點**:
  - ✅ **防呆效果最好**：路徑非常清晰，且極度適合在平板（Tablet）等觸控設備上橫向滑動操作。
  - ❌ **全局視野受限**：無法像節點圖那樣「一眼看到所有關聯」。

## 🛠️ 建議的實施策略 (Recommendation)

考慮到您的用戶是「中國的小型餐廳」，並希望未來發展為 SaaS，我強烈建議**優先選擇方案二（樹狀表格）+ 行內編輯**，並在未來 SaaS 版本將**方案一（可視化工作流）**作為高級/付費功能。

**近期可立即優化的 UX 行動：**
1. **取消「新增明細」的彈窗 (Dialog)**：改為在配方列表底部固定一行「新增行」，選完材料、敲入數字按 Enter 即保存，極大提升錄入效率。
2. **直觀的成本佔比條**：在配方詳情中，加入一根彩色的長條圖（類似硬碟空間佔比），一眼看出這道菜是肉花錢還是調料花錢。
3. **半成品防呆**：如果一個半成品配方（如滷肉汁）正在被多個商品使用，修改它時，必須彈出警告：*「此修改將影響 5 個依賴此半成品的商品成本，是否確認？」*
