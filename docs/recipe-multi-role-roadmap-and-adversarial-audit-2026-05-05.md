# 配方多角色 Roadmap 审计与对抗性审计

最后更新：2026-05-05

## 背景

当前系统已经具备配方、成本、库存、菜单的基础能力，但“能用”不等于“好用”。本审计从多角色视角和对抗性视角出发，评估配方链路在真实门店环境中的可执行性与抗误操作能力。

## 审计对象

- 原材料注册
- 批次与成本基线
- 半成品建模
- 成品配方建模
- 菜单绑定
- 类型管理
- 删除与批量编辑

## 多角色审计

### 1. 店长 / 经营者

关注点：

- 菜品是否赚钱
- 改价后毛利是否变化
- 新品上线速度是否够快
- 删除一个类型或配方会不会影响既有菜单

当前问题：

- 菜单绑定与配方成本关系不够直观
- 看得到单位成本，看不到成本来源
- 不能快速批量调整配方类型、启停状态

路线建议：

- 优先增加“绑定菜单后毛利预览”
- 在配方详情展示已绑定菜单与影响范围
- 加入配方批量状态切换

### 2. 厨务 / 研发 / 后厨主管

关注点：

- 半成品怎么产出
- 一个成品用了哪些材料和半成品
- 损耗率和单位是否设置正确

当前问题：

- 半成品没有专门工作流，只能和普通配方混着建
- 产出物、产出单位、产出状态没有完整前台表达
- 成本展开能看，但业务语义不够强

路线建议：

- 增加“半成品配方”专用创建路径
- 强化产出字段建模和 UI 校验
- 在明细中强调“原材料 / 子配方 / 产出结果”三层关系

### 3. 采购 / 仓管

关注点：

- 成本从哪里来
- 批次变动后会不会影响配方成本
- 原材料录入是否规范

当前问题：

- 配方页没有明确标注成本使用的是哪类基础数据
- 示例数据和真实数据之间没有明显边界
- 材料与批次页和配方页之间跳转不顺

路线建议：

- 在配方成本侧显示单位、损耗、成本来源提示
- 给示例数据增加更清晰的说明
- 后续加入从材料页跳转到“被哪些配方使用”

### 4. 财务 / 成本控制

关注点：

- 成本口径是否统一
- 半成品成本有没有正确传递到成品
- 删除或修改是否会静默影响历史核算

当前问题：

- 缺少成本 lineage 视图
- 依赖存在时虽然已有部分提示，但还不完整
- 缺少操作留痕与批量改动风险预览

路线建议：

- 增加“子配方成本如何汇总到成品”的可视化
- 对依赖型修改统一弹出影响提醒
- 后续补操作日志或审计记录

### 5. 实施顾问 / 培训人员

关注点：

- 能否用一条标准路径教会门店
- 示例数据是否稳定可复现
- 页面结构是否足够清晰

当前问题：

- 配方页之前过于拥挤，不利于培训
- 示例配方按钮之前不稳定
- 类型管理与配方编辑混在一起时认知负担大

路线建议：

- 已改成 `Tabs` 分区，后续继续强化引导文案
- 示例配方按钮改为稳定的局部刷新
- 文档继续按照“角色 + 路径”方式维护

### 6. 数据管理员 / 系统维护者

关注点：

- 主数据是否一致
- 类型是否能治理
- 删除是否安全

当前问题：

- 系统多处仍然缺少统一的删除与依赖说明
- 批量编辑能力不足
- 某些历史模块仍存在中英混杂或繁简混杂风险

路线建议：

- 统一删除规范
- 增加批量治理入口
- 持续清理文案与主数据策略

## UX 方向的总体优化建议

这类系统最值得优化的，不是单个控件，而是业务流里的认知切换成本。

### A. 把“下一步”做成显式动作

应该让用户在每个节点都看到明确下一步：

- 菜单产品无配方：加入配方清单
- 配方无明细：开始完善
- 配方有明细但未绑定菜单：去绑定菜单

这比在页面上堆更多说明文字更有效。

### B. 把“状态”做成统一语言

建议把跨页面状态统一成少量术语，例如：

- 待完善
- 待核对
- 可出品
- 已绑定菜单

统一语言可以降低培训成本，也方便后续做筛选和批量操作。

### C. 把“依赖关系”做成随处可见

用户最怕的是不知道改了会影响谁。
因此后续 UX 应优先补：

- 这个配方被几个菜单使用
- 这个半成品被几个成品引用
- 这个类型下有几个活跃配方

### D. 减少跨页跳转造成的断层

当前问题不是缺页面，而是页面之间没有接力。
理想状态是：

- 菜单页发现缺配方，可以直接送到配方页并开始完善
- 配方页发现缺成本，可以跳到批次或材料
- 材料页发现被引用，可以回跳到相关配方

### E. 对高风险动作做分级提示

不是所有确认框都一样重要。
建议分级：

- 普通编辑：轻提示
- 影响多个配方成本：强提示
- 删除被引用对象：阻止并展示依赖

这才是业务系统里的真实 UX。

## 对抗性审计

以下场景不是“正常使用”，而是故意或高压下的真实误操作场景。

### A. 重复点击 / 重复提交

风险：

- 连点创建配方
- 连点示例配方按钮
- 网络慢时重复提交

现状：

- 新增配方已加入提交中状态
- 示例配方按钮本轮补了局部刷新和防重复点击

后续：

- 继续为批量动作补 loading 与幂等保护

### B. 旧数据 + 新数据混用

风险：

- 数据库里已有旧配方、旧菜单、旧批次
- 种子逻辑用固定 ID 写入，导致静默错绑

现状：

- 本轮已修示例配方，改为按编码查真实 ID，并自动回绑对应菜单

后续：

- 继续清理其他依赖固定 ID 的种子逻辑

### C. 删除仍被引用的对象

风险：

- 删除仍在使用的配方类型
- 删除仍被菜单绑定的配方
- 删除仍被配方引用的材料

现状：

- 配方类型已加使用中拦截
- 配方删除与部分编辑已有确认

缺口：

- 还缺统一依赖视图
- 还缺更明确的失败原因展示

### D. 单位与损耗输入错误

风险：

- 用户以为输入百分比，系统按小数存
- 材料单位和配方单位不一致

现状：

- 配方页损耗率显示与写入已修正为百分比心智

缺口：

- 还没有做更强的单位一致性校验

### E. 数据契约漂移

风险：

- 前后端 payload 结构变了，前台直接白屏

现状：

- 本轮已修复 `RecipeWithItems` 契约不一致
- 前端新增 payload normalize 兜底

后续：

- 需要继续减少“隐式数据结构约定”

### F. 批量操作缺失导致高频重复劳动

风险：

- 人员通过重复单条编辑修改几十个对象
- 过程中高概率改错、漏改

现状：

- 菜单有部分批量能力
- 配方和材料基本还没有

后续：

- 批量编辑是接下来必须补的管理能力，不是锦上添花

### 里程碑 2：引导式工作流

- 区分半成品与成品创建入口

- 强化菜单绑定说明

### 里程碑 3：依赖与成本透明

- 增加依赖视图

## 建议执行顺序

### 第 1 阶段：动作化

先补最直接影响体验的上下文动作：

- 菜单无配方：补配方
- 配方无明细：开始完善
- 配方未绑定菜单：去绑定菜单

原因：

- 这是最低成本、最高体感收益的改造
- 不依赖复杂数据结构调整

### 第 2 阶段：状态化

把流程状态统一成前端可计算的状态系统：

- 待完善
- 待核对
- 可出品
- 已绑定菜单

原因：

- 后续筛选、批量、提醒都要依赖这层

### 第 3 阶段：依赖透明

增加“谁影响谁”的可见性：

- 菜单绑定数
- 子配方引用数
- 类型使用数
- 材料被使用数

原因：

- 不做这层，删除和批量操作会一直危险

### 第 4 阶段：跨页接力

把菜单、材料、批次、配方串成业务流：

- 菜单 -> 配方
- 配方 -> 菜单
- 配方 -> 批次 / 材料
- 材料 -> 配方

原因：

- 这是让系统从“后台页集合”升级成“工作流系统”的关键

### 第 5 阶段：批量治理

最后再做批量，因为它依赖前面几层：

- 明确动作
- 明确状态
- 明确依赖

没有前面三层，批量编辑只会把风险放大。
